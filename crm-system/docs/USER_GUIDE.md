## 一、系统启动

### 第一步：启动后端

打开终端，执行：

```bash
cd D:\develop\job\telegram-bot\crm-system\backend
uvicorn app.main:app --reload --port 8080
```

看到 `Uvicorn running on http://127.0.0.1:8080` 即启动成功。

### 第二步：浏览器访问

打开浏览器访问 **http://localhost:8080**

> **注意**: 前端代码修改后需要重新构建 `cd frontend && npm run build`。后端 `--reload` 会自动检测 Python 代码变化。

## 三、消息管理

消息是订单的原始来源。消息管理的入口在侧边栏 **消息管理**。

### 3.1 导入消息

**准备 JSONL 文件**：从 Telegram 群聊导出的消息文件，每行一个 JSON 对象。

支持两种来源：
- **历史消息 (dump)** — 通过 `telegram-bot-demo/dump_messages.py` 拉取的历史消息
- **实时捕获 (live)** — 通过 `telegram-bot-demo/live_capture.py` 实时捕获的消息

**操作步骤**：
1. 点击 **导入消息** 按钮
2. 选择消息来源（历史/实时）
3. 点击 **选择文件**，选取 `.jsonl` 文件
4. 点击 **开始导入**
5. 界面提示"导入 N 条，跳过 M 条"

**去重机制**：以 `(群组ID, 消息ID)` 为唯一键，重复导入会自动跳过。

### 3.2 查看消息

导入后消息列表显示：
- **群名称** — 消息来源群组
- **发送者** — 发送人名称
- **消息内容** — 悬停可看完整内容
- **时间** — 消息发送时间
- **处理状态** — 已处理/未处理

可以通过"处理状态"下拉框筛选未处理消息。

### 3.3 提取订单（关键操作）

从消息中自动提取订单有两种方式：

**方式 A：单条提取**
- 在消息列表中点击某条消息的 **处理** 按钮
- 如果匹配提取规则 → 自动生成订单，提示"提取成功"
- 如果未匹配 → 提示"未匹配到任何规则"

**方式 B：批量提取**
- 点击 **批量提取** 按钮
- 确认对话框中点确定
- 系统自动处理所有未处理消息，提示"处理 N 条，生成 M 个订单"

## 四、订单管理

### 4.1 订单生命周期

```
消息提取 → 待审核 → 已分配 → 处理中 → 已完成
              ↓       ↓        ↓        ↓
            已驳回   已取消    已取消    已取消
```

### 4.2 管理员操作

**创建订单**（手动）：
1. 进入 **订单管理** 页面
2. 点击 **新增订单**
3. 填写订单编号（必填）、客户信息、产品、金额、区域等
4. 提交 → 状态为 `待审核`

**分配订单**（手动或自动）：
- **自动分配**：审核员审核通过时，系统自动按规则分配给业务员
- **手动分配**：点击订单详情中的 **分配** 按钮，选择业务员

**更新订单**：点击订单行进入详情，修改字段后保存。

### 4.3 审核员操作

审核员侧边栏有单独的 **待审核** 菜单。

1. 进入 **待审核** 页面
2. 点击订单行，查看详情
3. **审核通过** → 订单状态变为 `已分配`，触发自动分配
4. **审核驳回** → 填写驳回原因 → 状态变为 `已驳回`

### 4.4 业务员操作

1. 进入 **订单管理**，仅看到分配给自己的订单
2. 点击订单进入详情
3. **开始处理** → 状态变为 `处理中`
4. **完成订单** → 状态变为 `已完成`
5. **取消订单** → 状态变为 `已取消`（仅在处理中状态可取消）

### 4.5 筛选与搜索

订单列表支持：
- 按状态筛选（待审核、已分配、处理中、已完成等）
- 按关键字搜索（订单编号或客户名称）
- 分页浏览

## 六、规则配置

> 仅 admin 可访问。入口在侧边栏 **规则配置**。

### 6.1 提取规则

提取规则决定从消息中识别哪些内容、生成什么订单字段。

规则页面分两个 Tab：

**提取规则 Tab**：

| 规则属性 | 说明 |
|----------|------|
| 名称 | 规则显示名称 |
| 类型 | `keyword` / `regex` / `sender` / `group` |
| 规则配置 | JSON 格式，每种类型格式不同 |
| 优先级 | 数字越大越先执行 |
| 启用状态 | 开关控制 |

**各类型配置示例**：

```json
// keyword — 消息包含关键词则命中
{"keywords": ["下单", "订购", "来货"]}

// regex — 正则匹配并提取字段
{"pattern": "[¥￥](\\d+(?:\\.\\d{1,2})?)", "field_mapping": {"total_amount": 1}}

// sender — 来自指定发送者则命中
{"sender_ids": [123456789, 987654321]}

// group — 来自指定群组则命中
{"group_ids": [-100123456]}
```

**操作**：
- **新增规则**：点击"新增规则"按钮 → 填写表单 → 保存
- **编辑规则**：点击某行"编辑" → 修改配置 → 保存
- **启用/禁用**：开关切换

### 6.2 分配规则

控制审核通过后订单分配给哪个业务员。

**分配规则 Tab**：

| 规则属性 | 说明 |
|----------|------|
| 类型 | `region`（按区域分配）/ `load_balance`（负载均衡） |
| 规则配置 | JSON 格式 |

**配置示例**：

```json
// 按区域分配 — 华东订单分配给列表中第一个可用的 agent
{"region_id": 1, "agent_ids": [3]}

// 负载均衡 — 华东订单分配给当前最闲的 agent
{"region_id": 1}
```

**分配逻辑**：
1. 审核员审核通过时自动触发
2. 按优先级从高到低匹配规则
3. `region` 类型：region_id 匹配 → 分配给 agent_ids 中的第一个可用 agent；未指定 agent_ids 则自动负载均衡
4. `load_balance` 类型：查询该区域所有 agent，选择当前订单数最少的
5. 如无匹配规则，订单保持 `已分配` 状态但未分配处理人，由 admin 手动分配

## 八、典型工作流

### 完整流程（从消息到完成）

```
步骤 1: 导入消息
  [消息管理] → 导入消息 → 选择 JSONL 文件 → 开始导入

步骤 2: 配置规则 (首次)
  [规则配置] → 提取规则 → 新增 "订单触发词" (keyword: ["下单","订购"])
  [规则配置] → 分配规则 → 新增 "华东默认分配" (region: {"region_id":1})

步骤 3: 批量提取
  [消息管理] → 批量提取 → 确认 → 生成订单

步骤 4: 审核
  [review 登录] → 待审核 → 选中订单 → 审核通过

步骤 5: 处理
  [agent1 登录] → 订单管理 → 选中订单 → 开始处理 → 完成订单

步骤 6: 验证
  [admin 登录] → 订单管理 → 查看订单状态 = "已完成"
```

### 手动创建订单流程（跳过消息提取）

```
步骤 1: 创建订单
  [admin 登录] → 订单管理 → 新增订单 → 填写信息 → 提交

步骤 2-6: 同上（审核 → 分配 → 处理 → 完成）
```

## 补充说明

### 关于消息来源

消息来自 Telegram 群聊，拉取脚本在 `D:\develop\job\telegram-bot\telegram-bot-demo\` 目录：

- `dump_messages.py` — 历史消息批量导出，适用于首次导入
- `live_capture.py` — 实时消息捕获，持续运行，适用于生产环境

两个脚本均输出 JSONL 文件，可导入本系统。

### 关于去重

消息导入以 `(group_id, message_id)` 作为唯一标识。同一消息重复导入会被自动跳过，不会生成重复订单。

### 关于自动分配

自动分配在审核通过时触发。如果已手动分配了处理人，自动分配不会覆盖。如果无匹配分配规则，订单保持 `已分配` 状态，需手动分配。
